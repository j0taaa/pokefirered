## Field Script Parity Tracker

Updated: 2026-04-23

Closed this pass:
- `special` / `specialvar` coverage is now explicitly tracked in the field-script interpreter.
- The browser port now has a concrete handler classification for every decomp `special` and `specialvar` referenced by the loaded script sources, with a regression test that fails if a new one appears without coverage.
- `waitstate` now suspends on the same browser-port field tasks that open deferred UI/message flows, instead of being treated like a no-op after PC menus, list menus, save prompts, and other supported async field interactions.
- Several item-reward and simple single-player subsystem NPCs that were still handwritten stubs now run through their original decomp scripts again, including Viridian/Pewter mart clerks, the Fishing Guru, the Bike Shop voucher exchange, the Fan Club chairman, the VS Seeker woman, and gym-leader reward branches like Misty / Lt. Surge.
- `pokemart` now runs through a dedicated FireRed-style shop state instead of the old field `listmenu` shortcut, including the original buy/sell top menu loop, quantity selection, yes/no confirmation, money updates, sell-value handling, and the matching canvas shop windows.
- In-game trades, daycare party-selection/retrieval flows, move tutor selection, and move relearner selection now run through the decomp field scripts instead of handwritten stubs or auto-picked browser-port shortcuts. The interpreter now resolves the original trade/daycare/tutor specials, list-driven party choices, and Cape Brink tutor flag logic directly from decomp data.
- Link / Union Room / Wireless Club / Direct Corner field flows now use the original cable-club scripts instead of browser-port stubs. The browser port now resolves the real offline/wired failure branches for the attendants, cable-club menus, save prompts, and adapter checks, so those NPCs behave like FireRed does when multiplayer connectivity is unavailable.
- Basic decomp object movement is now interpreted instead of skipped. `applymovement` can run real movement blocks for the player and NPC object events, including walking, in-place facing, face-player, face-away-player, face-original-direction, delay steps, and emote steps used by common cutscene scripts.
- Door and warp field commands are now tracked as runtime state. `opendoor`, `closedoor`, `waitdooranim`, `setdynamicwarp`, `setwarp`, `warp`, and `warpspinenter` now mirror the decomp script flow closely enough for cutscenes and elevators to remember door state, dynamic warp targets, and pending scripted map transitions.
- Camera-object and visual/cutscene field specials now leave explicit runtime side effects instead of disappearing as pass-throughs. `SpawnCameraObject` / `RemoveCameraObject` can run decomp camera movement blocks, teleporter animation phases, screen shake, Pokemon League lighting, S.S. Anne / Seagallop ferry scene triggers, and fall warps are recorded in runtime state, and pass-through specials are counted in `fieldEffects.triggeredSpecials`.
- Dialog message-mode commands now follow the decomp state model more closely. `messageautoscroll` opens auto-scroll field messages, `signmsg` / `normalmsg` persist and clear signpost-frame mode across following `message` / `msgbox` commands, and braille script text is parsed from `.braille` blocks and displayed through the braille message path.
- Field message rendering now has a decomp-style text-printer state. Dialogue pages reveal over time using the original slow/mid/fast frame delays from `new_menu_helpers.c`, rendered text uses the visible printer prefix instead of the full page immediately, and the page-advance arrow only appears after the printer finishes.
- A/B input now routes through the active field text printer before page advancement. Once printing has begun, pressing A/B requests the same speedup path as `RenderText`, held A/B keeps the printer delay at zero, and only later confirmation advances the page/script after the printer is complete.
- Field choice/list windows now clamp to the decomp multichoice bounds (`left + width <= 28 tiles`) and the GBA viewport height, preventing variable-sized NPC dialog menus from rendering off-screen.

Remaining high-level gaps:
- No known high-level browser-port field-script command-flow gaps remain for NPC dialogue/cutscene execution. Remaining dialog-system work is now narrow renderer fidelity: exact glyph metrics/tile blitting and full text control-code behavior.
